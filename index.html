Codex 项目化使用指南

知识库 · AGENTS.md · Skills · Plugins / MCP：把 Codex 从“代码聊天框”配置成项目智能体工作台。

核心结论：Codex 不应只当“会写代码的聊天框”，而应当配置成项目智能体工作台。推荐结构是：项目资料库负责知识来源，AGENTS.md 负责长期工作规则，Skills 固化重复任务流程，Plugins / MCP 负责连接外部工具和资料源。

目录

1. Codex 与 NotebookLM 的差异 2. 建立项目知识库的三种方式 3. 用 AGENTS.md 管理 Codex 的工作方式 4. Skills 的作用、结构与创建方法 5. Plugins / MCP 与 Skills 的区别 6. 技术与 AI 原理：为什么这些配置有效 7. 适合你的落地路线 8. HTML 博客发布方案 9. 推荐阅读与官方资料

1. Codex 与 NotebookLM 的差异

NotebookLM 的核心是“先上传资料，再围绕资料问答”；Codex 的核心是“围绕项目目录、代码、文档、工具和任务执行”。所以在 Codex 中，更推荐把知识库做成“项目资料库 + 检索规则 + MCP/RAG 工具”的形式，而不是只把资料堆进聊天窗口。

维度NotebookLMCodex 推荐方式
主要对象文章、PDF、网页资料项目目录、代码、文档、脚本、测试、外部工具
核心工作基于资料进行问答和摘要理解项目并执行修改、测试、分析、生成文档
知识来源上传的资料集合docs/knowledge、AGENTS.md、Skills、MCP、外部搜索
适合场景文献阅读、资料问答、知识整理代码工程、项目开发、自动化流程、复杂任务执行

因此，Codex 的“知识库”应该和项目工程结构绑定：把可复用资料放入项目目录，让 AGENTS.md 告诉 Codex 何时查资料、优先查什么、如何区分资料证据和模型推理。

2. 建立项目知识库的三种方式

2.1 轻量级：本地文档库

这是最适合起步的方法。把资料整理为 Markdown / TXT / HTML，并放到项目的 docs/knowledge/ 目录下。AGENTS.md 负责说明资料索引和使用规则。

my-project/
├─ AGENTS.md
├─ docs/
│  ├─ knowledge/
│  │  ├─ paper_notes/
│  │  ├─ official_docs/
│  │  ├─ tutorials/
│  │  └─ project_decisions.md
│  ├─ glossary.md
│  └─ sources_index.md
├─ .codex/
│  ├─ config.toml
│  └─ skills/
│     └─ literature-review/
│        └─ SKILL.md
├─ scripts/
│  └─ search_knowledge.py
└─ src/
  • docs/knowledge/：放文章、教程、论文笔记、接口文档、项目决策。
  • docs/sources_index.md：记录资料来源、更新时间、可信度、适用范围。
  • AGENTS.md：规定 Codex 遇到相关问题时先查哪些资料。
  • Skills：把“如何查资料、如何引用、如何综合分析”固化成可复用流程。
建议：不要直接把大量 PDF 原文堆给 Codex。更稳妥的做法是把核心资料转成 Markdown，并按主题拆分，便于模型定位和引用。

2.2 中量级：本地 RAG 检索脚本

当资料达到几十篇以上时，单靠 AGENTS.md 指引可能不够。可以建立本地 RAG：资料切分为 chunk，向量化存入 FAISS/Chroma，再根据问题检索最相关片段，让 Codex 基于片段分析。

资料 → 切分 chunk → embedding 向量化 → 向量库 FAISS/Chroma
    → 根据问题检索相关片段 → Codex 读取片段并分析
python scripts/search_knowledge.py "YOLOv11 小目标人员检测 mosaic 应该怎么调"

2.3 高级：MCP 知识库或外部工具接入

MCP 适合把 Codex 连接到外部工具或共享资料源，例如 Google Drive、GitHub、Context7、本地 RAG 服务、浏览器上下文等。

codex mcp add context7 -- npx -y @upstash/context7-mcp

3. 用 AGENTS.md 管理 Codex 的工作方式

AGENTS.md 可以理解成“给项目智能体看的 README”。它适合放项目目标、目录结构、关键命令、禁止事项、测试方法、知识库入口和完成标准。

层级路径示例作用
全局个人规则~/.codex/AGENTS.md所有项目都继承的个人偏好和工作习惯
项目规则项目根目录/AGENTS.md当前项目的目标、目录、命令、禁止事项
子模块规则src/detection/AGENTS.md某个子目录或模块的特殊规则，优先级更高

3.1 全局 AGENTS.md 示例

# Terry 的 Codex 全局工作规则

## 交流方式
- 默认使用中文回答。
- 遇到复杂任务，先给计划，再执行。
- 不要一上来大改项目结构，先说明影响范围。
- 修改代码前先阅读相关文件，确认现有逻辑。
- 输出命令时，给出完整可复制命令。

## 编程习惯
- 尽量保留原有代码风格。
- 修改前说明改哪些文件。
- 修改后给出测试命令。
- 如果不确定依赖版本，先检查环境文件。
- 不要擅自删除数据、模型权重、训练结果、日志文件。

## 任务完成标准
- 代码能运行。
- 给出验证方法。
- 说明可能的风险。
- 说明下一步怎么做。

3.2 项目 AGENTS.md 示例

# 本项目 Codex 工作规则

## 项目背景
这是一个 YOLOv11 人员检测与状态识别项目，目标是在边缘端统计人员停留时间。

## 重要目录
- datasets/：数据集与标注
- scripts/：数据处理脚本
- runs/：训练结果，不要删除
- models/：模型定义
- docs/knowledge/：项目知识库

## 修改规则
- 修改训练脚本前，先检查 data.yaml、模型权重路径、batch、imgsz、冻结层和学习率。
- 不要覆盖原始训练结果。
- 新实验必须使用新的 name 参数。
- 任何会改变标签顺序的操作必须先提醒用户确认。

## 回答要求
- 涉及训练参数时，必须说明为什么这么调。
- 涉及报错时，必须先定位根因，再给命令。
- 涉及部署时，必须区分 Python 推理、ONNX、TensorRT、C++ 部署。
AGENTS.md 的边界：AGENTS.md 不适合放大量论文原文、长篇教程、完整接口文档。它更适合放规则、索引、命令、禁止事项和完成标准。

4. Skills 的作用、结构与创建方法

一句话区分：AGENTS.md 管“这个项目里 Codex 应长期遵守什么规则”；Skill 管“遇到某类任务时，Codex 应按什么流程做”。

对象作用典型内容
AGENTS.md长期工作规则项目目录、测试命令、禁止事项、知识库入口
Skill可复用任务流程任务触发条件、步骤、输出格式、脚本、参考资料
Script确定性执行工具检查日志、生成报告、搜索知识库、格式转换

一个 Skill 通常是一个目录，里面必须有 SKILL.md，也可以带 scripts、references、assets。Codex 先看 name 和 description 判断是否相关；匹配后再读取完整 SKILL.md。

~/.codex/skills/
└─ yolo-training-review/
   ├─ SKILL.md
   ├─ scripts/
   │  └─ check_yolo_run.py
   ├─ references/
   │  └─ yolo_param_notes.md
   └─ assets/

4.1 YOLO 训练分析 Skill 示例

---
name: yolo-training-review
description: 当用户需要分析 YOLO 训练参数、训练日志、mAP、loss 曲线、过拟合、欠拟合、数据集问题或微调策略时使用。
---

# YOLO 训练分析流程

## 第一步：收集信息
先检查：
- data.yaml
- 训练脚本
- 模型权重路径
- epochs、batch、imgsz
- lr0、lrf、optimizer
- mosaic、mixup、copy_paste
- freeze
- results.csv
- confusion_matrix、PR 曲线、样例预测图

## 第二步：判断问题类型
区分：
- 数据量不足
- 标签错误
- 学习率过低
- 冻结层不合理
- 增强策略不匹配
- 类别不平衡
- 小目标分辨率不足
- 过拟合或欠拟合

## 第三步：输出建议
必须给出：
- 当前问题判断
- 修改参数
- 完整训练命令
- 风险说明
- 如何验证是否改善

5. Plugins / MCP 与 Skills 的区别

Skill 是工作流程；Plugin 是可安装的能力包；MCP 是外部工具或资料接口；App Connector 是连接 Gmail、Google Drive、GitHub 等具体服务的桥。

概念本质适合解决的问题
Skill流程 / 方法论 / 操作手册让 Codex 按固定步骤处理某类任务
Plugin可安装能力包把多个 Skills、MCP、App 映射和资源打包分发
MCP外部工具 / 资料接口协议连接第三方文档、数据库、本地服务、浏览器、Figma 等
App Connector具体应用连接连接 Google Drive、GitHub、Slack、Gmail 等
my-plugin/
├─ .codex-plugin/
│  └─ plugin.json
├─ skills/
│  └─ my-skill/
│     └─ SKILL.md
├─ hooks/
│  └─ hooks.json
├─ .mcp.json
├─ .app.json
└─ assets/
使用建议：现阶段不要一上来做 Plugin。先做 Skill，因为 Skill 更容易调试。等某个 Skill 稳定后，再考虑打包成 Plugin；如果需要连接外部资料或工具，再接 MCP。

6. 技术与 AI 原理：为什么这些配置有效

Skills 和 AGENTS.md 并不是给模型重新训练，也不是安装了一个新模型。它们本质上是“上下文工程”和“任务路由规则”：让模型在合适的时候加载合适的规则、流程、资料和工具。

任务输入
↓
Codex 读取全局/项目 AGENTS.md
↓
根据任务判断是否需要某个 Skill
↓
读取对应 SKILL.md、references、scripts
↓
必要时调用 MCP/脚本/外部搜索
↓
基于项目上下文、资料证据和模型推理输出结果
  • 减少重复提示词：把反复使用的工作方法固化，不必每次重新描述。
  • 提高触发一致性：description 负责让模型判断何时使用某个 Skill。
  • 节省上下文：Codex 不需要一次性读取所有 Skill，而是按任务逐步加载。
  • 增强可验证性：脚本、测试命令、输出格式可以把不稳定的口头经验变成可检查流程。
  • 降低跑偏概率：AGENTS.md 明确禁止事项、目录约束、测试标准和完成标准。

7. 适合你的落地路线

  1. 每个重要项目先建立 docs/knowledge/、docs/sources_index.md、AGENTS.md。
  2. 在 AGENTS.md 中写清项目背景、目录结构、知识库位置、代码修改规则、测试命令和禁止事项。
  3. 先做两个高频 Skills：yolo-training-debug 和 paper-translation-polish。
  4. 每次发现 Codex 反复犯同一个错误，就把规则补进 AGENTS.md 或对应 Skill。
  5. 当资料变多后，让 Codex 帮你写 scripts/build_knowledge_index.py 和 scripts/search_knowledge.py。
  6. 当知识库、Google Drive、GitHub、网页资料都需要接入时，再考虑 MCP。
  7. 当某套能力稳定后，再打包为 Plugin 供多个项目或团队复用。

8. HTML 博客发布方案

本次同时生成了 HTML 文件，可以作为静态网页直接上传。后续如果希望互联网可访问，推荐优先选择 GitHub Pages；如果你想以后扩展成更漂亮的个人博客或前端站点，可选 Vercel 或 Netlify。

方案适合人群优点注意事项
GitHub Pages只想快速公开 HTML / Markdown 文档的人免费、稳定、适合静态页面；支持自定义域名需要 GitHub 账号和仓库；国内访问速度视网络而定
Vercel希望后续扩展成博客、Next.js、AI 工具站的人部署快、适合现代前端项目、Git 集成方便功能较多，纯小白初次可能略复杂
Netlify想拖拽或 Git 自动部署静态站的人静态站部署简单，适合 HTML/CSS/JS国内访问速度同样受网络影响
自有服务器 + Nginx已有服务器和域名的人完全可控，可放到自己的域名下需要维护服务器、HTTPS、备份和安全配置
最推荐路线：先用 GitHub Pages 发布这个 HTML。后续如果你需要更像博客系统的能力，再迁移到 Vercel + Astro/Next.js，或 Netlify + 静态站点生成器。
