一、总体判断：Obsidian 适合做 Codex 的长期知识库，但不要把它当 NotebookLM

Obsidian 和 Codex 的组合非常适合长期项目，但它的定位不是“把资料上传进去问答”，而是“把项目知识长期沉淀为可维护的 Markdown，再让 Codex 按规则、流程和工具去检索与使用”。

最佳理解：Obsidian 负责沉淀知识；AGENTS.md 负责告诉 Codex 怎么工作；Skills 负责把重复流程标准化；搜索脚本、RAG 或 MCP 负责让 Codex 找到相关资料。

Obsidian 的笔记本质上是本地 Markdown 文件，Vault 是本地文件夹，因此它天然适合被 VS Code、Git、Codex、脚本和 RAG 工具读取。Obsidian 的内部链接和图谱视图对人类整理知识很有帮助，但它本身不是语义检索系统。

工具最适合承担的角色不适合承担的角色
Obsidian长期知识库、项目笔记、决策记录、错误记录、博客草稿自动化执行器、完整代码代理
AGENTS.md项目规则、目录说明、命令、约束、完成标准大段资料库、论文原文、网页合集
Skills重复任务流程，如调参分析、报错排查、文档整理无限制的百科知识库
MCP / RAG连接外部工具、语义检索大量资料、调用文档源替代知识整理本身

二、推荐架构：Obsidian 是长期记忆，Codex 项目是执行现场

推荐把知识库和代码项目分开管理，再用复制、目录联接、脚本检索或 MCP/RAG 连接起来。这样既不会污染代码仓库，也能让知识跨项目复用。

五、第三步：把 Obsidian 接到 Codex 项目里

根据项目复杂度，可以选择三种连接方式。

方案 A：复制精选资料

把 Obsidian 中最相关的几篇复制到项目的 docs/knowledge/。适合刚开始，最稳定。

方案 B：目录联接

用 Windows Junction 把 Obsidian 的项目笔记挂载到 Codex 项目的 docs/obsidian-kb/。

方案 C：搜索脚本

项目里不复制笔记，只放搜索脚本，让 Codex 按规则检索 Vault。

方案 D：RAG/MCP

当笔记数量很大时，把 Obsidian 转为语义检索服务，再通过 MCP 暴露给 Codex。

方案 B 的 Windows 示例

cd D:\Projects\codex-guide
New-Item -ItemType Junction -Path docs\obsidian-kb -Target "D:\AI_Knowledge\Codex_KB_Projects\Codex使用体系"
注意：如果使用 Git 管理 Obsidian Vault，不建议提交 .obsidian/workspace.json 这类频繁变化的工作区文件。可以把它加入 .gitignore。

六、第四步：在 Codex 项目里写 AGENTS.md

AGENTS.md 是 Codex 进入项目后要先读的“项目工作说明书”。它不应该放大量资料，而应该放规则、路径、约束、测试命令和完成标准。

# Codex 项目工作规则

## 项目背景

本项目使用 Obsidian 作为长期知识库，用 Codex 进行项目分析、代码修改、文档整理和知识沉淀。

## 重要目录

- docs/knowledge/：项目内精选知识资料
- docs/obsidian-kb/：从 Obsidian Vault 挂载或复制过来的项目知识
- docs/handoff/：历史对话迁移、任务交接、决策记录
- scripts/：用于检索知识库、整理笔记、生成文档的脚本
- .agents/skills/：项目级 Codex Skills

## 使用知识库的规则

当任务涉及以下内容时，必须先查知识库：

- 项目历史决策
- 技术选型
- 报错排查
- 训练参数
- 部署流程
- 论文/PPT/文档写作规则
- 用户已经总结过的工作方法

优先级：

1. 当前项目 AGENTS.md
2. docs/handoff/
3. docs/knowledge/
4. docs/obsidian-kb/
5. 外部官方文档
6. 搜索结果或模型推理

## 回答时必须区分信息来源

回答中需要明确区分：

- 项目知识库已有结论
- 当前代码或文件体现的信息
- 外部资料/官方文档信息
- Codex 自己的推理判断

## 修改代码前的要求

- 先阅读相关文件，不要直接重写
- 不要删除历史实验结果、模型权重、数据集、日志
- 修改前说明涉及哪些文件
- 修改后给出可复制的验证命令
- 如果不确定，先给计划，不要直接大改

## 任务结束后的知识沉淀

每次完成重要任务后，需要询问是否更新：

- docs/handoff/task-log.md
- docs/handoff/decisions.md
- docs/handoff/known-issues.md
- Obsidian 对应项目笔记

## 完成标准

一个任务完成时，至少说明：

1. 做了什么
2. 修改了哪些文件
3. 如何验证
4. 还有什么风险
5. 是否需要更新知识库

七、第五步：创建一个 Obsidian 知识库检索 Skill

AGENTS.md 管项目长期规则，Skill 管某类任务的固定流程。建议创建一个 obsidian-context-loader，让 Codex 在遇到历史资料、项目决策、报错记录、教程资料时先查知识库。

my-codex-project/

   └─ skills/
      └─ obsidian-context-loader/
         └─ SKILL.md

---

# Obsidian 知识库检索流程

## 目标

帮助 Codex 在回答或修改项目之前，先从 Obsidian 知识库中找出相关历史资料，避免重复分析和遗漏已有结论。

## 第一步：判断是否需要检索

以下情况必须检索：

- 用户提到“之前”“上次”“已有资料”“知识库”“Obsidian”
- 任务涉及项目决策、部署、训练参数、报错记录
- 用户要求综合多篇文章、资料、教程
- 任务需要延续另一个 Codex 项目或历史对话

以下情况可以不检索：

- 简单语法解释
- 单个短命令修改
- 用户明确要求不要查历史资料

## 第二步：检索位置

优先搜索：

- docs/handoff/
- docs/knowledge/
- docs/obsidian-kb/
- D:\AI_Knowledge\Codex_KB_Projects- D:\AI_Knowledge\Codex_KB_Topics
排除：

- .obsidian/
- 90_Attachments/
- 00_Inbox/ 未整理资料

## 第三步：检索方法

先使用关键词搜索，必要时扩大同义词。

例如：

- “YOLO 训练” 同时搜索：yolo、训练、mAP、loss、freeze、mosaic
- “Codex 知识库” 同时搜索：codex、AGENTS、Skills、MCP、RAG、Obsidian
- “部署” 同时搜索：ONNX、TensorRT、trtexec、C++、推理

## 第四步：读取和筛选

只读取最相关的 3-8 篇笔记。

筛选标准：

1. 更新时间较新
2. 与当前项目直接相关
3. 来源可靠
4. 有明确结论或命令
5. 不是临时草稿

## 第五步：回答要求

回答时必须包含：

- 查到了哪些相关笔记
- 这些笔记支持什么结论
- 哪些内容仍需外部查询或重新验证
- 当前建议
- 是否需要把新结论写回知识库

八、第六步：先用 ripgrep 搜索，不要一开始就上复杂 RAG

刚开始不要急着做向量数据库。先把目录、模板、标签和文件命名做好，再用关键词搜索建立工作习惯。

安装 ripgrep

winget install BurntSushi.ripgrep.MSVC

搜索整个 Obsidian Vault

rg -n --glob "*.md" --glob "!**/.obsidian/**" "Codex AGENTS Skills" "D:\AI_Knowledge\Codex_KB"

搜索某个项目

rg -n --glob "*.md" "YOLO freeze 学习率 mosaic" "D:\AI_Knowledge\Codex_KB_Projects\YOLO人员检测"
升级到 RAG/MCP 的条件：Markdown 超过几百篇、关键词搜索经常找不到同义表达、你希望用自然语言自动召回相关笔记，或者多个项目都需要共享同一个知识库。

RAG 的基本流程是：Markdown → 切分 chunk → embedding 向量化 → 向量库 → 根据问题检索相关片段 → Codex 读取片段再结合项目文件分析。MCP 则可以把这个检索能力包装成 Codex 能调用的工具。

九、Codex + Obsidian 的日常工作流

任务开始：先让 Codex 阅读 AGENTS.md，并用 obsidian-context-loader 搜索相关笔记。
执行之前：要求 Codex 先输出已有资料、已有结论、风险和执行计划，不要马上修改代码。
执行过程中：Codex 修改代码、运行测试、记录报错，并与知识库中的历史方案对照。
任务结束：整理 task note、decision note、error note，把有长期价值的结论写回 Obsidian。
每周整理：清理 00_Inbox，合并重复笔记，把成熟流程转成 Skill。

任务开始提示词

请先阅读 AGENTS.md。
然后使用 obsidian-context-loader 检索 Obsidian 知识库中和“YOLOv11 人员检测、小数据集微调、freeze、学习率、mosaic”相关的笔记。
先不要修改代码。
请先输出：
1. 找到的相关资料
2. 已有结论
3. 当前任务风险
4. 建议执行计划

任务结束提示词

请把这次任务整理成 Obsidian 笔记。
需要生成三份内容：

1. task note：记录本次做了什么、修改了什么、验证了什么
2. decision note：如果这次形成了长期技术决策，请写成决策记录
3. error note：如果遇到报错，请写成可复用排查记录

请使用项目知识库模板，输出为 Markdown。

十、是否把 Obsidian 知识库发布到互联网博客？

可以发布，但不建议全量发布。Obsidian 中可能包含项目路径、日志、账号线索、未验证结论、私人记录和商业资料。更安全的方式是分成私有知识库和公开博客库。

Codex_KB/
├─ 10_Projects/        私有，不发布
├─ 20_Topics/          部分可发布
├─ 50_Public_Blog/     专门发布
└─ 90_Attachments/     发布前筛选

如果只是发布几篇文章，GitHub Pages + 单文件 HTML 就足够。如果未来要把 Obsidian 做成长期公开知识花园，可以考虑 Quartz 或 Obsidian Publish。

十一、最适合的落地路线

阶段 1：建 Vault

先建立 D:\AI_Knowledge\Codex_KB 和基础目录，不追求自动化。

阶段 2：接入 Codex

写 AGENTS.md，把相关笔记复制或挂载到项目里，安装 rg 搜索。

阶段 3：沉淀 Skills

先做 obsidian-context-loader、knowledge-curator、project-handoff-writer。

阶段 4：升级 RAG/MCP

当笔记数量和项目数量变多后，再做语义检索和 MCP 服务。

最终建议：Obsidian 做长期知识库，Codex 项目里放 AGENTS.md，高频流程做 Skills，简单检索先用 rg，资料变多后再升级 RAG/MCP。这样比单纯 NotebookLM 式“上传资料问答”更适合 Codex，因为 Codex 还要读项目、改代码、跑命令、生成文档和更新知识库。
